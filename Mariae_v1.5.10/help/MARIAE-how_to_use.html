1. General

1.1 - How to Install and Run the Tool

  1. Download the latest version of the tool from its download page
  2. Unzip the downloaded file into a distribution folder (from now on, we call this folder as "mariae_home")
  3. Run the tool. There are two possible ways to do this:
    • (via command line) Open a shell, go to the mariae_home folder, then run the following command: "java -jar Maria.jar"
    • Alternatively, double-click on the "Maria.jar" file

1.2 - Explanation of MARIAE Menus

  • File - To hande the files in the tool
    • New.. - To create a new (AUI/CUI) model. When you select this menu item, a window will appear enabling you to select the type of UI model you want to create.
    • Open.. - To openan already existing model (e.g. a task model, an AUI model, a CUI model). Please note that UI models that are compliant with MARIA language typically have a ".xml" extension
    • Save - To save the updates done to the currently active model
    • Save as... - To save the current model and possibly change its name
    • Import ASFE-DL... - Converts an ASFE-DL abstract user interface definition into MARIA XML. The ASFE-DL is one of the markup languages submitted for the standardization for the W3C MBUI Working Group.
    • Close File - To close the current model
    • Exit - To exit the tool
  • Edit
    • Cut - To cut the current selection
    • Copy - To copy the current selection
    • Past - To past the current selection
  • Tool
    • CTT - To activate a number of functionalities for handling task models specified with ConcurTaskTrees (CTT) notation
      • Start Task Simulator - This feature activates the task model simulator, which enables you to interactively simulate the behaviour of the currently loaded task model. After you select this option, you will see the tasks that are enabled at the beginning of the simulation (which have been automatically derived according to the temporal relationships specified in the task model). Then, you can select one of such tasks (by double-clicking on it) and then the simulation will move to the next state, which is the new state reached after executing the task just selected. You can repeat this as many times as the task model specification enables. Please note that, depending on the task model at hand the simulation might even never stop (e.g. this could occur when after a number of steps you reach again the initial state of the simulation, or when the root of the task model is iterative, ..).
      • Generate AUI - This feature enables you to generate an Abstract UI model from the currently loaded CTT task model. When you activate such a functionality, the tool automatically transforms the CTT task model into an AUI model, by applying some default settings to carry out this step.
      • Generate AUI from Presentation Task Sets.. - Differently from the previous functionality in this case the user has more control on the transformation that generates an AUI model from a CTT task model. In particular, the user will be asked to select a number of heuristics to be used during this transformation. Such heuristics are typically aimed at reducing the number of AUI presentations that will be generated in the end by this transformation. If you do not want to bother with these heuristics you can directly use the Generate AUI menu item explained before to easily obtain an AUI model from a CTT task model
      • Generate Task Model from WSDL.. - This functionality enables you to obtain a CTT task model starting from a WSDL description of a Web Service
    • AUI - This enables you to activate a number of functionalities for handling AUI models compliant with the MARIA language
      • Generate Desktop CUI - You can apply this functionality to a currently loaded AUI specification. In this case, the tool automatically derives a CUI specification for the desktop platform
      • Generate Vocal CUI - You can apply this functionality to a currently loaded AUI specification. The tool automatically derives a CUI specification for the vocal platform
      • Generate Mobile CUI - You can apply this functionality to a currently loaded AUI specification. In this case, the tool automatically derives a CUI specification for the mobile platform
      • Generate MultiModal Desktop CUI - You can apply this functionality to a currently loaded AUI specification. The tool automatically derives a CUI specification for the multimodal desktop platform
      • Generate Multimodal Mobile CUI - You can apply this functionality to a currently loaded AUI specification. In this case, the tool automatically derives a CUI specification for the multimodal mobile platform
    • CUI Desktop - Under this menu you will find a group of functionalities that can be applied to a CUI specification for desktop platform (compliant with the MARIA language)
      • Generate HTML Preview - You can apply this functionality to a currently loaded CUI specification for the desktop platform. The tool will provide you with a preview of an HTML implementation UI generated starting from the input CUI model
      • Generate JSP with Web Service Support - This functionality can be applied to a currently loaded CUI specification for a desktop platform. The tool will generate a JSP page enhanced with a support for calling to Web Services
      • Generate SMIL - By applying this functionality to the currently loaded CUI specification for desktop, the tool will generate an implementation UI specified according to the SMIL language
      • Generate Mobile CUI - Differently from the above three functionalities, this feature does not implement a model-to-code transformation. Rather, it support a model-to-model semantic redesign transformation. By applying this transformation to a CUI desktop, you will obtain another CUI specification semantically redesigned and adapted for the mobile platform. This transformation uses some rules for performing this adaptation (for instance some interactors are replaced by other interactors more suitable for the mobile platform, a splitting step might be carried out for supporting a a more effective pagination of the original desktop presentations, ..)
    • CUI Mobile Large - This enables you to activate a number of functionalities for handling CUI models for the mobile platform (compliant with the MARIA language)
      • Generate HTML - To generate HTML implementation code from the currently loaded CUI specification for the mobile-large platform
    • CUI Vocal - This enables you to activate a number of functinalities for handling CUI models for the mobile platform (compliant with the MARIA language)
      • Generate Voice XML 2.0 - To generate an implementation UI specified according to the Voice XML 2.0 language
      • Grammar Generator - To generate a grammar that will be used for the recognition of the vocal UI
    • CUI Multimodal - This enables you to activate a number of functionalities for handling CUI models for the multimodal platform (compliant with the MARIA language)
      • Generate XHTML+Voice - To generate an implementation UI specified according to XHTML and Voice XML 2.0 languages
      • Change CARE Values - CARE (Complementarity, Assignment, Redundancy, Equivalence) properties are aimed to manage the different modalities that can occur in a multimodal interactive application. For instance, in a graphical+vocal multimodal application you might want to specify that the same interaction should be supported both in a graphical manner AND in a vocal way: in this case you should then assign redundancy value to that interaction. This menu option provided in the tool enables you to handle such properties.
    • Transformation Editor - It activates a tool module enabling you to specify the rules for performing model-to-model transformations (e.g. transforming an AUI model into a CUI model).

      To use this feature you will be asked to specify at least:

      • a source model (e.g. an AUI model)
      • a target model (e.g. a CUI model),
      • a set of rules to be applied for carrying out the transformation. With MARIAE such rules can be either selected by loading a file (in which such rules could have been saved previously), or they could be defined by graphically specifying the mappings (by graphically drawing lines connecting the UI elements in the comcerned models).
    • Transformation by XSLT - It enables you to directly specify the XSLT rules to be applied to carry out a transformation
    • Apply XSLT... - It enables you to directly select an XSLT file and immedialely see the result. The procedure will ask you what type of model (AUI, CUI desktop, mobile etc.) the transformation will produce.
  • Projects - In this menu you can find a number of features for handling Projects in the tool
    • New Model.. - To start creating a new UI model
    • Import Existing Model.. - To import an existing UI model
    • New Project.. - To create a new project
    • Open Project.. - To open an existing project
    • Select Main Project.. - When you have multiple projects loaded in the tool, you may want to set the currently main one. This feature enables you to do this.
    • Close Main Project - To close the current main project
    • FTP Settings - To set FTP setting (this could be needed for instance to load some files from/to a FTP server)
  • Help - Some help information
    • User's guide.. The current guide
    • About.. - Information about the development team/organisation (HIIS Lab of ISTI-CNR, Pisa Italy)

2. Handling Projects in the Tool

MARIAE is a tool for handling a number of multi-layer (task, abstract, concrete, implementation layer) UI descriptions of your interactive applications. Using MARIAE you will end up creating a number of UI models referring to the same scenario/example. For this reason, in order to easily handle the files referring to the same example/application/scenario, we provided you with the possibility to collect such models/files within the same project. Please note that MARIAE projects generally have a ".meprj" extension.

2.1 - How to Create a New Project

  1. Select Projects|New Project, then browse your hard disk to select the folder where you want to create a new project
  2. In the file chooser window that will appear, provide a name for your project (e.g. "museumApplication_project", "educational_scenario", ..)
  3. An empty project folder will be automatically created in the tool (see the Project tab in the left-hand part of the tool), with a set of nested sub-folders (e.g. ctt, aui, cui, annotations, etc.) that will contain each a different type of content. Correspondingly, a file directory with the same nested structure will be also created in your hard disk. At this point you can start populating your empty project by e.g. importing a UI model in your project

2.2 - How to Open an Existing Project

  • Actions to do

    Select Projects | Open Project.., then browse your hard disk to select an existing project (they generally have a ".meprj" extension). Then, a feedback message "Project loaded" will appear. Press "OK" and in the left-hand panel of the tool the folder of the project will be shown. Double-click on its icon to open it and be able to browse and see its content.

  • Try an example

    Select Projects | Open Project , then, within the mariae_home folder, go to the "examples" folder and then open the "homeScenario.meprj" file. The associated project will be loaded and visualised within the tool.

2.3 - How to Set the Main Project

Within MARIAE you could have multiple projects loaded at the same time (they are all listed when you select the “Projects” tab). In case of several projects loaded in the tool, you can easily identify the current main project, since the associated folder name is shown in bold letters within the tool (if just one project is loaded, by default this project is also the main project). All the files generated from a AUI/CUI will be saved in the main project.

  • Actions to do

    In the left-hand panel of MARIAE, right-click on the folder of the project that you want to elect as the new main project, then select "Set as main project”.

  • Try an example

    To see how this feature works on a concrete example, do the following:

    1. Load (at least) two projects in MARIAE:
      • From the tool menubar select Projects | Open Project , then select (and load) the "homeScenario.meprj" project from the examples folder. After, select (and load) the "weatherScenario.meprj".
    2. Identify the main project:
      • In the left-hand panel of the tool only one project's name will be shown in bold: this is the current main project
    3. Change the main project to another one:
      • Right-click on the folder name of the other project (the one different from the current main one), then select Set as main project, to set it as the new main project.

3. Handling MARIA UI models (in General)

Please note tha the UI models that are handled in the tool are the ones compliant with the specification of the MARIA language

3.1 - How to Create a New AUI model/CUI Model

Select File |New in the menubar. Then, in the window that will appear, double-click on type of UI model you want to create.

3.2 - How to Open a UI Model in the Tool

In the tool menubar select File | Open.. and then browse your hard disk and select the file corresponding to the model you want to open in the tool (e.g. a task model, an AUI model, ..). The content of the selected file will be opened and then rendered in the central panel of the tool.

3.3 - How to Import a UI Model within the Main Project

At a certain point you might want to import an existing model within your current main project load inthe tool (please note that the files corresponding to CTT task models, Abstract User Interface (AUI) models, and Concrete User Interface (CUI) models, will generally have a ".xml" extension).

  • Actions to do
    1. From the tool menubar, select Projects | Import existing Model. Then, browse your hard disk and select the file containing the model to be imported.
      • Note: depending on the information contained in the file you select for importing, the tool will automatically include that file within the appropriate folder of the current main project. This means that, e.g. if you select a file corresponding to a CTT task model to be imported in the current project, the tool will automatically include such file into the "ctt" folder of the current project; if you import a file that corresponds to an Abstract User Interface (AUI) description, the importing action will put this file within the "aui" folder of the current main project, and so on also for the other types of UI models.
    2. The selected file/model will be imported and loaded in the current main project . This means that:
      • the content of the imported file will be shown in the central panel of the tool
      • the concerned model will be included in the appropriate folder of the current main project
  • Try an Example

    As a concrete example, let's try to import a CTT task model in your current main project.

    1. Let's indicate <taskModel_path> as the path of the CTT task model to be imported (e.g. in Windows-based OSs, <taskModel_path> could be something like "C:/Windows/users/mariae_127/../ctt/my_taskmodel.xml") .
      • Note: if you do not have a CTT task model available (please note that no task model is provided "ready-made" within the MARIAE distribution folder), you can easily build a CTT task model in two possible ways:
        • either by building a CTT task model by using the CTTE environment
        • or, by using MARIAE, you could try the example provided in the section "How to generate a CTT task model from a WSDL". After having run this example, you will have a CTT task model named "HomeScenarioWSversion2Service.xml" and saved in the "examples/homeScenario_files/ctt" folder
    2. From the menubar, select "Projects | Import existing Model. Then, browse your hard disk and select the file named examples/homeScenario_files/ctt/HomeScenarioWSversion2Service.xml.
    3. As a consequence, the corresponding CTT task model will be imported in the main project, which means that:
      • the content of the imported CTT file will be graphically rendered in the main (central) panel of the tool
      • the CTT file will be included in the "ctt" folder of the current main project (see the left-hand panel of the "Projects" tab panel)

4. Handling CTT Task Models

4.1 - How to Generate a CTT Task Model from a WSDL

  • Actions to do

    Select Tool.. | CTT | Generate Task Model from WSDL . From the pull-down menu that will appear, select the service from which you want to derive the CTT task model, then press "OK" in the dialog that will appear. The associated task model will be automatically generated, saved, and shown within the tool.

  • Try an example

    Select Tool.. | CTT | Generate Task Model from WSDL. From the pull-down menu that will appear, select the service called “HomeScenarioWSVersion2Service”, then press “OK”. The generated task model will be automatically shown (and saved in the appropriate folder of the tool).

4.2 - How to Select and Apply Heuristics to Obtain an AUI

Select Tools | CTT | Generate AUI from Presentation Task Sets. A window will open allowing you to select the heuristics to be applied to the Presentation Task Sets that have been automatically identified starting from the analysis of the task model. Apply as many heuristics as you want. When you have finished, click the "Finish" button: the AUI will be automatically generated and the first presentation of such AUI will be automatically displayed in the main panel of the tool.

  • Note: Presentation Task Sets are groups of tasks that are enabled at the same period of time. Presentation Task Sets are identified according to information contained in the task model (e.g. temporal relationships between tasks, ..) The identification of Presentation Task Sets is needed in order to derive an AUI model from a task model. For more information on Presentation Task Sets please refer to this paper.

4.3 - How to Obtain an AUI from a CTT Task Model

From the tool menubar, under the Tool | CTT menu, select the Generate AUI menu item

  • Note: If the menu item Generate AUI is graied, this means that the feature is not active since in the current project there is no task model available, to which apply this feature (e.g. no task model file appears in the “ctt” folder of the current main project). So, in order to use this feature please first ensure that you have loaded a task model in the tool. If not, either try to import a task model in the current main project (see Section "How to import a model in the main project") or just load a CTT task model in the tool (see section How to open a UI Model in the Tool).

5. Handling AUI models

5.1 - How to Modify an AUI

Select the "Document" tab of the vertical tabbed panel shown in the left part of the tool: the list of the various concrete UI presentations will be displayed. By clicking on one of such presentation in the left part of the window, the content of the selected presentation will be displayed in the central panel of the tool. You can refine/customize such shown presentation in at least two possible ways:

  • Select the "Design" tab (shown in the left-bottom corner of the central panel), which enables you to handle the shown model by graphically acting on its design structure. Then, within this view, fold/unfold UI elements and/or right-click/double-click on the various interactors in order to edit their properties.
  • Alternatively, select the "XML" tab (shown in the left-bottom corner of the central panel), which enables you to directly modify the UI model by acting on its XML-based description. Please note that, under this "XML" view you can have further functionalities that should help you while editing the model (for instance, you can easily search the occurrence of a certain interactor).

Notes:

  • When you are under the "Design" view and you have currently selected a certain interactor, you can look up at its corresponding XML-based specification (described according to the MARIA language) by just selecting the "XML" tab.
  • Please note that the changes done under one of the above views (Design or XML) will be automatically reflected in the other view)

5.2 - How to Add a New AUI Element into an AUI Presentation

When you have an AUI and you would like to add a new AUI element element in one of its abstract UI presentation, you could this in at least two manners:

  • Either select the element from the right-hand panel of the tool and then drag-and-drop it within the central panel of the tool (where the presentation in which you want to add this element is currently shown)
  • Or right-click in the central panel of the window, in correspondence of where you want to add the concerned element, and then select the item "Add new" from the context-sensitive menu that will appear

5.3 - How to Change the Attributes of an AUI Interactor

In order to change the attributes/properties of a certain AUI interactor included in an AUI model currently open/loaded in the tool (and then shown in the central panel of the window) you can do this in two different manners:

  • Either double-click on the concerned AUI element whose properties you want to edit. A new window will appear where you can edit the abstract properties of the concerned element.

  • Right-click on the concerned AUI element whose properties you want to edit and then select "Quick Edit". A new window will appear where you can edit the abstract properties of the concerned element.

5.4 How to obtain a CUI from an AUI

From the tool menubar, under the Tool | AUI menu, select the proper item depending on the type of platform you want to address. For instance, please select Generate Desktop CUI if you want to obtain a CUI description for the desktop platform, select Vocal CUI if you want to generate a CUI for the vocal platform, and so forth for the other platforms.

6. Handling CUI Models

6.1 - How to Modify a CUI

Select the "Document" tab of the vertical tabbed panel shown in the left part of the tool: the list of the various concrete UI presentations will be displayed. By clicking on one of such presentation in the left part of the window, the content of the selected presentation will be displayed in the central panel of the tool. You can refine/customize such shown presentation in at least two possible ways:

  • Select the "Design" tab (shown in the left-bottom corner of the central panel), which enables you to handle the shown model by graphically acting on its design structure. Then, within this view, fold/unfold UI elements and/or right-click/double-click on the various interactors in order to edit their properties.
  • Alternatively, select the "XML" tab (shown in the left-bottom corner of the central panel), which enables you to directly modify the UI model by acting on its XML-based description. Please note that, under this "XML" view you can have further functionalities that should help you while editing the model (for instance, you can easily search the occurrence of a certain interactor).

Notes:

  • When you are under the "Design" view and you have currently selected a certain interactor, you can look up at its corresponding XML-based specification (described according to the MARIA language) by just selecting the "XML" tab.
  • Please note that the changes done under one of the above views (Design or XML) will be automatically reflected in the other view)

6.2 - How to Add a New CUI Element into a CUI Presentation

If you have a CUI model and you would like to add a new element in one of its CUI presentations, you could this in at least two possible manners:

  • Select the element from the right-hand panel of the tool and then drag-and-drop it within the central window (where the presentation in which you want to add this interactor is currently shown)
  • Alternatively, right-click in the central part of the window, in correspondence of where you want to add the element, and then select the item "Add new" from the context-sensitive menu that will appear

6.3 - How to Change Attributes of a CUI Interactor

In order to change the attributes/properties of a certain CUI interactor included in a CUI model currently open/loaded in the tool (and then shown in the central panel of the window) you can do this in two different manners:

  • Double-click on the concerned CUI element whose attributes/properties you want to edit. A new window will appear where you can edit the properties of the concerned element.
  • Alternatively, right-click on the concerned CUI element whose properties you want to edit and then select "Quick Edit". A new window will appear where you can edit the properties of the concerned element.

In particular, such window will enable you to edit two kinds of attributes associated with a certain CUI interactor:

  • Abstract attributes: they are the interactor's attributes that have been inherited from the abstract level
  • Concrete attributes: they are the interactor's attributes which are specific of the concrete UI level, and then refer to the specific platform at hand

6.4 - How to Obtain a Final UI (FUI) from a CUI Model

With this feature you can obtain an implementation UI starting from a CUI for a certain platform (for instance, you can generate HTML code from a CUI Desktop, or you can generate VoiceXML2.0 for a CUI vocal, ..). In order to do this, from the tool menubar, under the Tool menu, select the proper menu item depending on the type of platform for which you want to build an UI implementation (e.g. CUI Desktop, CUI Mobile Large,.). As you will see, various platforms are currently supported by the tool (desktop, mobile large, vocal, multimodal).

7. How to run the scenarios

7.1 How to run the home and weather scenario

The mariae_home folder is the location where you created unzipped the MARIAE package downloaded from the HIIS lab web site.

The scenarios demostrate the tool capability to generate applications from annotated WSDL definitions. The annotation files and the addresses of the WSDL definitions is contained into the sample projects. The selected scenario (homeScenario or weatherScenario) will be referenced as scenario

  1. Open the scenario.meprj project file in the mariae_home/examples folder from the menu Projects | Open Project... .
    When the project is loaded, set it as the main one, right-clicking its icon on the Project Panel and selecting Set as Main Project
  2. Generate the task model for the WSDL of the scenario web service from the menu Tools | CTT | Generate Task Model from WSDL. Select the HomeScenarioWSversion2 for the home scenario or GlobalWeather for the weather scenario from the drop down list. The generated CTT is automatically opened in the environment and saved into the current project. (You can browse the Web service definition through the WS panel on the right.
    3. It is possible to open other WSDL definition using the Web Services menu).
  3. Generate the abstract user interface (AUI) for the CTT generated at step 2 by selecting Tools | CTT | Generate AUI The generated AUI is automatically opened in the environment and saved into the current project
  4. Generate the desktop concrete user interface (desktop CUI) for the AUI generated at the step 3 by selecting Tools | AUI |Generate Desktop CUI
  5. Generate the final web application (Java Servlet + JSP) for the desktop CUI generated at step 4 by selecting Tools | CUI Desktop | Generate JSP with Web Service Support. It will create a NetBeans 6.9 web project that can be compiled and deployed on a Java Servlet and JSP server (e.g. Apache Tomcat). The project is located into the mariae_home/Output/generated folder. Open it from NetBeans and, after having configurated the server, you can run the application directly from the Netbeans IDE by right-clicking the project icon in the project panel and selecting Run.
  6. The home scenario application can be accessed with username guest and password user.

7.2 How to run the educational scenario

The mariae_home folder is the location where you created unzipped the MARIAE package downloaded from the HIIS lab web site.

The scenario demostrates the tool capability to generate applications from a CTT task model bound to specific Web Service operations. The task model has been created by a designer and it is contained into the CTT project folder. The operations have been annotated in advance by the service creator. The annotation file and the address of the WSDL definition is contained into the sample project.

  1. Open the educational.meprj project file in the mariae_home/examples folder from the menu Projects | Open Project... .
    When the project is loaded, set it as the main one right-clicking its icon on the Project Panel and selecting Set as Main Project
  2. Open the task model double clicking the educational-scenario-v-3.xml contained into the CTT node from the project panel on the left. The CTT model is automatically opened in the environment and you can browse the Web service definition through the WS panel on the right.
  3. Generate the abstract user interface (AUI) for the CTT generated at step 2 by selecting Tools | CTT | Generate AUI The generated AUI is automatically opened in the environment and saved into the current project. It is also possible to fine-tune the AUI generation by manually applying the heuristics selecting Tools | CTT | Generate AUI from Presentation Task Sets
  4. Generate the desktop concrete user interface (desktop CUI) for the AUI generated at the step 3 by selecting Tools | AUI |Generate Desktop CUI
  5. Generate the final web application (Java Servlet + JSP) for the desktop CUI generated at step 4 by selecting Tools | CUI Desktop | Generate JSP with Web Service Support. It will create a NetBeans 6.9 web project that can be compiled and deployed on a Java Servlet and JSP server (e.g. Apache Tomcat). The project is located into the mariae_home/Output/generated folder. Open it from NetBeans and, after having configurated the server, you can run the application directly from the Netbeans IDE by right-clicking the project icon in the project panel and selecting Run.
  6. The home scenario application can be accessed with username 54321 and password spano.

7.3 How to run the car rental scenario

The mariae_home folder is the location where you created unzipped the MARIAE package downloaded from the HIIS lab web site.

The scenario demostrates the tool capability to generate applications a WADL service definition. Please note that the message exchange is supported only with XML serialization. The address of the WADL definition is contained into the sample project. Here we will list the steps for generating a vocal application for renting a car that exploits the REST web service. In addition, we demonstrate the MARIAE ability to convert the ASFE-DL abstract user interface definition into MARIA XML. The ASFE-DL is one of the markup languages submitted for the standardization for the W3C MBUI Working Group.

However, it is also possible to automatically generate a default application for the other platforms following the same steps listed for the home and weather scenarios.

  1. Open the carRentalScenario.meprj project file in the mariae_home/examples folder from the menu Projects | Open Project... .
    When the project is loaded, set it as the main one right-clicking its icon on the Project Panel and selecting Set as Main Project
  2. Import the ASFE-DL AUI definition selectingFile | Import ASFE-DL . From the Open file dialog, select the carRent.xml file in the mariae_home/examples/carRentalASFE_DL folder.
    The tool will automatically load the MARIA XML version of the AUI, and will save it into the current project.
  3. Generate the vocal concrete user interface (vocal CUI) from the AUI generated at the step 2 by selecting Tools | AUI |Generate Vocal CUI The tool will automaticall open the Vocal CUI and will save it into the current project.
  4. Now we will add a set of platform-dependent information to this CUI, applying an XSLT to the current displayed Vocal CUI.
    Select Tools | Apply XSLT... and select the PopulateVocalCUI.xsl file in the mariae_home/examples/carRentalASFE_DL folder.
    The tool will prompt you asking what kind of model the transformation will produce. Select cui vocal. After that, the tool will automatically load another version of the same VocalCUI file with more information, and it will save it into the current project.
  5. Generate the final web application (Voice XML + Java Servlet + JSP) for the vocal CUI generated at step 4 by selecting Tools | CUI Vocal | Generate VoiceXML 2.0 with Web Service Support. It will create a NetBeans 6.9 web project that can be compiled and deployed on a Java Servlet and JSP server (e.g. Apache Tomcat). The project is located into the mariae_home/Output/generated folder. Open it from NetBeans and, after having configurated the server, you can run the application directly from the Netbeans IDE by right-clicking the project icon in the project panel and selecting Run. You can browse the vocal application through the Voxeo Vocal Browser.

7.4 How to run the multimodal scenarios

The multimodal scenario project file is located in maria_home/examples/multimodalScenario.meprj. The scenario shows how it is possible to generate an X+V application from a multimodal CUI. In order to generate the application you should follow the following steps:

  1. Open the multimodalScenarioi.meprj project file in the mariae_home/examples folder from the menu Projects | Open Project... .
    When the project is loaded, set it as the main one right-clicking its icon on the Project Panel and selecting Set as Main Project
  2. Open the multimodal-desktop CUI or multimodal-mobile CUI (Multimodal_Desktop_flight_example) available in the Project tab.
  3. Generate the X+V implementation of the model selecting Tools | CUI Multimodal | Generate XHTML + Voice . At the end of the transformation, a popup will specify the exact path of the generated files on your machine (by default they will be in the mariae_home/examples/multimodalScenario_files/x+v/ folder).
    Open the transformation results with an X+V enabled browser (e.g. Opera max version 9.27)
  4. Generate the Multimodal HTML implementation Tools | CUI Multimodal | Generate Multimodal HTML . At the end of the transformation, a popup will specify the exact path of the generated files on your machine (by default they will be in the mariae_home/examples/multimodalScenario_files/MultimodalHtml/ folder).
    The new multimodal user interface generator produces HTML implementations structured into two parts: one for the graphical and one for the vocal interface.
    Such implementations are supported in different ways depending on the device type, desktop or mobile.
    In case of desktop applications we exploit the Web Speech API (https://dvcs.w3.org/hg/speech-api/raw-file/tip/speechapi.html) which defines 2 interfaces: Speech Recognition and Speech Synthesis.
    The Speech Recognition interface is currently supported by Google Chrome v25 or later.
    The Speech Synthesis interface is, at this moment, under development and to exploit it users have to download the latest Google Chrome version for develpers that it is named Google Chrome Canary and it is available at https://www.google.com/intl/en/chrome/browser/canary.html.
    Google Chrome Canary has the newest of the new Chrome features, it's designed for developers and early adopters, and can sometimes break down completely.
    Because of the Speech Synthesis interface is under development users have to enable it from Chrome settings:
    • Go to chrome://flags page
    • Find the row 'Exprerimental Web Platform features'
    • Enable this feature
    • Restart the browser
    Regarding the mobile platform the multimodal support is has been implemented only for Android devices: to open the new Multimodal HTML transformation users have to download the HIIS Multimodal Application at http://giove.isti.cnr.it/tools/MultimodalHtml/home
